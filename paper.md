## Abstract

Current AI agent skill systems use static markdown files as prompts. We propose **computed skills** — an architecture where a program analyzes context (git state, conversation history, past outcomes, structured data) and generates a tailored markdown prompt for each invocation. The skill file becomes a thin shell that calls the program; the real intelligence lives in code that decides *what to instruct*. We demonstrate this pattern on two production skills in a personal AI agent, showing that Python-powered skills eliminate redundant LLM token consumption on structured data parsing, enable per-invocation strategy adaptation, and introduce persistent feedback loops that static prompts cannot support.

## 1. Introduction

AI coding agents (Claude Code, Cursor, Windsurf, Aider) use markdown-based skill systems to extend their capabilities. A skill is typically a `SKILL.md` file containing instructions the LLM follows when the skill triggers. This architecture is elegant — markdown is human-readable, version-controllable, and native to LLMs.

But static skills have a fundamental limitation: **the same instructions fire regardless of context.** A code review skill gives identical guidance whether you changed 2 auth files or 40 config files. A learning-capture skill forces the LLM to parse hundreds of structured entries every session to check for duplicates.

We introduce **computed skills** — a pattern where the skill's instructions are generated by a program at invocation time. The program:

1. Analyzes the current context (git diff, codebase state, file types, conversation patterns)
2. Consults persistent state (past invocations, outcomes, recurrence counts)
3. Selects a strategy (which lenses to apply, what depth, what to prioritize)
4. Generates a tailored markdown prompt

The LLM still receives markdown — its native instruction format. But the markdown is *computed*, not *written*.

## 2. Related Work

### 2.1 Context Engineering

Anthropic describes context engineering as "the art and science of filling the context window with just the right information for the next step" [1]. Their patterns — just-in-time retrieval, metadata-driven navigation, structured memory persistence — address what information to *retrieve*. Computed skills address what instructions to *generate*.

### 2.2 DSPy: Programmatic Prompting

Stanford's DSPy framework treats prompts as compilable programs [2]. Users define signatures (input/output behavior); optimizers find optimal instructions and few-shot examples. A 2025 study found DSPy-optimized prompts outperformed hand-crafted ones by up to 38% on specific tasks [3].

DSPy optimizes a *fixed* prompt through iterative compilation. Computed skills generate a *different* prompt per invocation based on live context. The approaches are complementary — one could use DSPy to optimize the templates within a computed skill.

### 2.3 Skill Systems

Claude Code's skill system [4] supports dynamic content injection via shell commands (`!`command``). Matthew Honnibal published well-crafted Python skills for Claude [5]. Lee Han Chung reverse-engineered the skill architecture, noting that skills are "specialized prompt templates" with progressive loading [6]. Martin Fowler observed that "almost all forms of AI coding context engineering ultimately involve a bunch of markdown files with prompts" [7].

These describe the status quo: static markdown with optional data injection. No prior work treats the *entire skill* as a program's output.

### 2.4 Self-Improving Agents

Research on self-improving AI agents focuses on fine-tuning, RLHF, and constitutional AI methods [8]. These operate at the model level. Computed skills enable self-improvement at the *prompt* level — the skill program tracks outcomes and adjusts its generated instructions accordingly, without model modification.

## 3. Architecture

### 3.1 Traditional Skill

```
SKILL.md (static markdown) ──→ LLM
```

The LLM receives the same instructions every time. Any context-dependence requires the LLM to read, parse, and reason about raw data — consuming tokens on bookkeeping.

### 3.2 Skill with Data Injection

```
SKILL.md (static markdown + !`command` placeholders) ──→ LLM
```

Shell commands inject data (e.g., `git diff --stat`), but the *instructions* around that data remain fixed. The LLM still receives the same review checklist whether the diff contains security files or config files.

### 3.3 Computed Skill

```
SKILL.md (thin shell) ──→ calls program
Program (analyzes context, consults state) ──→ generates markdown
Generated markdown ──→ LLM
```

The SKILL.md file reduces to a single line:

```yaml
---
name: smart-review
description: Context-aware code review
---

!`python3 ${CLAUDE_SKILL_DIR}/scripts/generate.py $ARGUMENTS`
```

The program is the intelligence. The markdown is its output. The LLM receives tailored, pre-analyzed instructions.

### 3.4 Design Principles

**Separation of concerns.** Python handles structured data (parsing, indexing, counting, filtering, age calculations). The LLM handles judgment (is this a real bug? should this entry be promoted?).

**Transparent reasoning.** The generated prompt explains *why* it chose a strategy. The LLM and user both see the reasoning:

```markdown
## Review Strategy (auto-selected)
**Depth**: deep — Small change (2 files, ~15 lines)
**Signals detected:**
- [security] Security-sensitive files: src/auth/login.js
**Lens weights:**
- Safety ██████████░░░░░░░░░░ 50%
- Correctness ██████░░░░░░░░░░░░░░ 30%
```

**Persistent state.** A JSON state file tracks invocation history, strategies used, outcomes. This enables:
- Repeat detection (same strategy twice → inject variety)
- Outcome tracking (what did past reviews catch/miss?)
- Evolution over time (the skill gets better at picking strategies)

**Graceful degradation.** If the program fails, the LLM falls back to generic instructions. If there's no git repo, no state file, or no data to analyze, the output is still valid — just less targeted.

## 4. Implementations

### 4.1 Code Review (Computed)

**Problem:** A static code review skill gives identical instructions for 2-file auth changes and 40-file config overhauls. The LLM wastes tokens re-reading the same checklist and applying irrelevant lenses.

**Solution:** A Python program that:

1. **Reads git state** — staged, unstaged, or last commit; file count; additions/deletions
2. **Detects patterns** — security-sensitive paths, config-heavy changes, test-only changes, script/automation patterns, large refactors
3. **Weights review lenses** — Security: 50% for auth files, Consistency: 35% for config changes, Correctness: 60% for test-only changes
4. **Selects review depth** — deep (line-by-line) for < 3 files, architectural (forest over trees) for > 20 files
5. **Integrates past learnings** — reads Pattern-Keys from a learning log to remind the LLM of recurring codebase issues
6. **Tracks history** — state file records past reviews, enabling repeat-strategy detection

**Results from testing:**

| Change type | Static skill | Computed skill |
|---|---|---|
| 2 auth files | Generic 4-lens review | Deep review, Safety 50%, Security Alert section |
| 22 config files | Generic 4-lens review | Architectural review, Consistency 35%, config-audit focus |
| Run #3 same files | Same generic review | Adds "Fresh Eyes" pass to avoid blind spots |

### 4.2 Self-Improve (Computed)

**Problem:** A learning-capture skill manages 87 entries across 3 markdown files (LEARNINGS.md, ERRORS.md, FEATURE_REQUESTS.md). Every session, the LLM must:
- Parse ~1200 lines of structured entries
- Scan for matching Pattern-Keys before logging
- Count recurrences manually
- Find promotion candidates (recurrence >= 2)
- Identify stale entries for triage

This is structured data processing — exactly what programming languages excel at and LLMs waste tokens on.

**Solution:** A Python program with 5 modes:

| Mode | Trigger | What Python does | What LLM does |
|---|---|---|---|
| `always-on` | Every session | Parse all entries, build pattern-key index, compute next sequence numbers, surface alerts | Make judgment calls on what to log |
| `status` | Manual `/self-improve` | Full dashboard: counts, promotion queue, stale entries, cross-file duplicates | Present to user, execute chosen action |
| `check <key>` | Before logging | Exact + fuzzy Pattern-Key lookup, recurrence projections | Decide whether to increment or create |
| `drift` | Heartbeat cycle | Load SOUL.md principles, scan daily logs for keyword drift signals | Deeper semantic drift check |
| `triage` | Monthly cleanup | Age-calculate all entries, bucket into stale/very-stale | Decide resolve/won't_fix/keep per entry |

**Token savings (estimated):**

| Operation | Static (LLM tokens) | Computed (LLM tokens) | Savings |
|---|---|---|---|
| Parse 87 entries | ~3,500 | 0 (Python) | 100% |
| Pattern-Key lookup | ~500 per check | ~50 (pre-indexed list) | 90% |
| Promotion candidates | ~1,000 | ~100 (pre-filtered) | 90% |
| Stale entry detection | ~800 | ~80 (pre-calculated) | 90% |

The LLM still sees the *results* (promotion candidates, matching keys, stale entries) but doesn't perform the *computation*.

### 4.3 Check-Pattern (Derived Skill)

A natural extension: a hidden skill (`user-invocable: false`) that the agent invokes silently mid-conversation when about to log a new learning. The program performs exact matching, fuzzy matching (word-overlap scoring), and returns actionable instructions:

- **Exact match:** "Don't create new entry. Increment [existing entry] to count N. This will trigger auto-promote."
- **Fuzzy match:** "Similar keys exist: [list]. Use existing key if same pattern."
- **No match:** "Safe to create new entry."

This creates a **skill-to-skill pipeline** — the always-on self-improve skill references the check-pattern skill, which calls the same Python backend with a different mode.

## 5. When To Use Computed Skills

Not every skill benefits from computation. The decision depends on what the skill does:

| Use static markdown when | Use computed skills when |
|---|---|
| Instructions are stable and context-independent | Strategy should adapt based on live context |
| The skill is simple (< 50 lines) | The skill works with structured data (logs, entries, diffs) |
| Anyone should be able to edit it | The LLM wastes tokens parsing instead of thinking |
| Transparency matters most | You want memory across invocations |
| The "what to do" never changes | The "what to do" depends on "what's happening" |

## 6. Levels of Computation

We observe a spectrum of skill computation complexity:

**Level 0 — Static.** Pure markdown. Same instructions every time.

**Level 1 — Injected.** Static markdown with `!`command`` data injection. Instructions fixed, data varies.

**Level 2 — Computed (this paper).** Program generates entire prompt. Instructions, strategy, and emphasis vary per invocation. Persistent state across runs.

**Level 3 — Self-optimizing.** Program tracks outcomes (what the LLM caught/missed), adjusts its own templates. Prompts evolve based on feedback.

**Level 4 — Coordinated.** Multiple computed skills share state and invoke each other. The check-pattern + self-improve pipeline is a minimal example.

**Level 5 — Emergent.** Skills that generate *other skills* — a meta-skill that observes recurring patterns and creates new computed skills to address them.

We implemented Levels 2 and 4. Level 3 is structurally supported (the state files include `hits` and `misses` arrays) but not yet active. Levels 5 is speculative.

## 7. Limitations

**Reduced transparency.** A static SKILL.md is readable as-is. A computed skill requires running the program to see what the LLM will receive. We mitigate this by including a "strategy explanation" section in every generated prompt.

**Development complexity.** Writing a Python program is harder than writing markdown. The audience for skill creation shrinks from "anyone who can write" to "anyone who can code."

**Failure modes.** If the program crashes, the skill produces no output. Static skills always produce output. We recommend defensive programming and fallback prompts.

**Over-engineering risk.** Simple skills don't benefit from computation. Adding a Python backend to a 10-line style guide is pointless complexity.

**Platform coupling.** The `!`command`` syntax is specific to skill systems that support shell command injection. Not all agent frameworks offer this.

## 8. Future Work

**Outcome-based optimization.** The state files already track invocation history. Adding a feedback mechanism (user rates review quality, or the system detects whether bugs were caught) would enable Level 3 self-optimization.

**Cross-skill state sharing.** Currently each skill has its own state file. A shared state layer would enable richer coordination between computed skills.

**Benchmark comparison.** Rigorous comparison of token usage, review quality, and task completion rates between static and computed versions of the same skill.

**Framework-agnostic implementation.** Port the pattern beyond Claude Code's `!`command`` syntax to other agent frameworks (Cursor rules, Aider conventions, custom agent systems).

## 9. Conclusion

Static markdown skills are the right default — simple, transparent, accessible. But for skills that work with structured data, need to adapt strategy per invocation, or benefit from persistent state, computed skills offer a strictly more capable architecture.

The key insight is a clean separation: **code computes, markdown instructs, LLM judges.** Python handles what it's good at (parsing, indexing, filtering, state management). The LLM handles what it's good at (reasoning, judgment, natural language understanding). Markdown is the interface between them.

The `!`command`` syntax in existing skill systems is the mechanism that makes this possible today, without any platform changes. The pattern is implementable immediately in any agent framework that supports shell command injection in prompts.

We release our implementations as open source to encourage experimentation.
